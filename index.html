1. Introduction

An entry list is a list of entries, typically representing the contents of a form. An entry is a tuple consisting of a name (a scalar value string) and a value (either a scalar value string or a File).

These definitions currently live in the XHR spec as part of the FormData definition, but they should probably be moved to HTML.

The multipart/form-data format provides a way to encode an entry list, which, unlike the application/x-www-form-urlencoded format, can transmit file contents.

2. multipart/form-data serializing

A multipart/form-data boundary string is a byte sequence such that:

  • its length is greater or equal to 27 and lesser or equal to 70, and

  • it is composed by bytes in the ranges 0x30 to 0x39, 0x41 to 0x5A, or 0x61 to 0x7A, inclusive (ASCII alphanumeric), or which are 0x27 ('), 0x2D (-) or 0x5F (_).

The multipart/form-data boundary string being a byte sequence rather than a string is an unfortunate historical accident.

To generate a multipart/form-data boundary string, return an implementation-defined byte sequence which fullfills the conditions for boundary strings, such that part of it is randomly generated, with a minimum entropy of 95 bits.

Previous definitions of multipart/form-data required that the boundary string associated with a multipart/form-data payload not be present anywhere in the payload other than as a delimiter, although they allow for generating the boundary string probabilistically. Since this generation algorithm isseparate from a payload, however, it has to specify a minimum entropy instead. [RFC7578] [RFC2046]

If a user agent generates multipart/form-data boundary strings with a length of 27 and an entropy of 95 bits, given a payload made specifically to generate collisions with that user agent‚Äôs boundary strings, the expected length of the payload before a collision is found is well over a yottabyte.

To escape a multipart/form-data name with a string name, an optional encoding encoding (default UTF-8) and an optional boolean isFilename (default false):

  1. If isFilename is true:

    1. Set name to the result of converting name into a scalar value string.

    Otherwise:

    1. Replace every occurrence of U+000D (CR) not followed by U+000A (LF), and every occurrence of U+000A (LF) not preceded by U+000D (CR), in name, by a string consisting of U+000D (CR) and U+000A (LF).

  2. Let encoded be the result of encoding name with encoding, converted to a byte sequence.

  3. Replace every 0x0A (LF) bytes in encoded with the byte sequence `%0A`, 0x0D (CR) with `%0D` and 0x22 (") with `%22`.

  4. Return encoded.

The multipart/form-data chunk serializer takes an entry list entries and an optional encoding encoding (default UTF-8), and returns a tuple of a multipart/form-data boundary string and a list of chunks, each of which can be either a byte sequence or a File:

  1. Set encoding to the result of getting an output encoding from encoding.

  2. Let boundary be the result of generating a multipart/form-data boundary string.

  3. Let output chunks be an empty list.

  4. For each entry in entries:

    1. Let chunk be a byte sequence containing `--`, followed by boundary, followed by 0x0D 0x0A (CR LF).

    2. Append `Content-Disposition: form-data; name="`, followed by the result of escaping a multipart/form-data name given entry‚Äôs name and encoding, followed by 0x22 ("), to chunk.

    3. Let value be entry‚Äôs value.

    4. If value is a string:

      1. Append 0x0D 0x0A 0x0D 0x0A (CR LF CR LF) to chunk.

      2. Replace every occurrence of U+000D (CR) not followed by U+000A (LF), and every occurrence of U+000A (LF) not preceded by U+000D (CR), in value, by a string consisting of U+000D (CR) and U+000A (LF).

      3. Append the result of encoding value with encoding, converted to a byte sequence, to chunk.

      4. Append 0x0D 0x0A (CR LF) to chunk.

      5. Append chunk to output chunks.

      Otherwise:

      1. Assert: value is a File.

      2. Append `; filename="`, followed by the result of escaping a multipart/form-data name given value‚Äôs name with encoding and isFilename set to true, followed by 0x22 0x0D 0x0A (" CR LF), to chunk.

      3. Append `Content-Type: `, followed by the result of isomorphic encoding value‚Äôs type, to chunk.

      4. Append 0x0D 0x0A 0x0D 0x0A (CR LF CR LF) to chunk.

      5. Append chunk, followed by value, followed by the byte sequence 0x0D 0x0A (CR LF), to output chunks.

  5. Append the byte sequence containing `--`, followed by boundary, followed by `--`, followed by 0x0D 0x0A (CR LF), to output chunks.

  6. Return the tuple boundary / output chunks.

This matches the behavior Gecko will have once bug 1686765 is closed, modulo the non-standard handling of directories. Chromium‚Äôs behavior also seems to match, except that it doesn‚Äôt yet perform newline normalizations at the serialization layer, which will be fixed in issue 1167095. I haven‚Äôt yet checked WebKit.

The length of a multipart/form-data payload, given a list of chunks which can be either byte sequences or Files chunks, is the result of running the following steps:

  1. Let length be 0.

  2. For each chunk in chunks:

    1. If chunk is a byte sequence:

      1. Increase length by chunk‚Äôs length.

      Otherwise:

      1. Assert: chunk is a File.

      2. Increase length by chunk‚Äôs size.

  3. Return length.

To create a multipart/form-data readable stream from a list of chunks which can be either byte sequences or Files chunks, run the following steps:

  1. Let file stream be null.

  2. Let stream be a new ReadableStream.

  3. Let pull algorithm be an algorithm that runs the following steps:

    if file stream is null and chunks is not empty

    1. If chunks[0] is a byte sequence, enqueue a Uint8Array object wrapping an ArrayBuffer containing chunks[0] into stream.

    2. Otherwise:

      1. Assert: chunks[0] is a File object.

      2. Set file stream to the result of running chunks[0]'s stream method.

    3. Remove the first item from chunks.

    if file stream is null and chunks is empty

    1. Close stream.

    if file stream is not null

    1. Let read request be a new read request with the following items:

      chunk steps, given chunk

      1. If chunk is not a Uint8Array object, error stream with a TypeError and abort these steps.

      2. Enqueue chunk into stream.

      close steps

      1. Set file stream to null.

      error steps, given e

      1. error stream with e.

    2. Let reader be the result of getting a reader for file stream.

    3. Read a chunk from reader with read request.

  4. Let cancel algorithm be an algorithm that runs the following steps, given reason:

    1. If file stream is not null, cancel file stream with reason.

  5. Set up stream with pullAlgorithm set to pull algorithm and cancelAlgorithm set to cancel algorithm.

  6. Return stream.
