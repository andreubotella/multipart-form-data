1. Introduction

An entry list is a list of entries, typically representing the contents of a form. An entry is a tuple consisting of a name (a scalar value string) and a value (either a scalar value string or a File).

These definitions currently live in the XHR spec as part of the FormData definition, but they should probably be moved to HTML.

The multipart/form-data format provides a way to encode an entry list, which, unlike the application/x-www-form-urlencoded format, can transmit file contents.

2. multipart/form-data serializing

A multipart/form-data boundary string is a byte sequence such that:

  • its length is greater or equal to 27 and lesser or equal to 70, and

  • it is composed by bytes in the ranges 0x30 to 0x39, 0x41 to 0x5A, or 0x61 to 0x7A, inclusive (ASCII alphanumeric), or which are 0x27 ('), 0x2D (-) or 0x5F (_).

The multipart/form-data boundary string being a byte sequence rather than a string is an unfortunate historical accident.

To generate a multipart/form-data boundary string, return an implementation-defined byte sequence which fullfills the conditions for boundary strings, such that part of it is randomly generated, with a minimum entropy of 95 bits.

Previous definitions of multipart/form-data required that the boundary string associated with a multipart/form-data payload not be present anywhere in the payload other than as a delimiter, although they allow for generating the boundary string probabilistically. Since this generation algorithm isseparate from a payload, however, it has to specify a minimum entropy instead. [RFC7578] [RFC2046]

If a user agent generates multipart/form-data boundary strings with a length of 27 and an entropy of 95 bits, given a payload made specifically to generate collisions with that user agent‚Äôs boundary strings, the expected length of the payload before a collision is found is well over a yottabyte.

The multipart/form-data length of an entry list entries and a boundary string boundary, with an optional encoding encoding (default UTF-8) is the result of running these steps:

  1. Set encoding to the result of getting an output encoding from encoding.

  2. Let length be 6 + the length of boundary.

  3. For each entry in entries:

    1. Let name length be the length of the result of escaping a multipart/form-data name given entry‚Äôs name and encoding.

    2. Let entry length be 49 + the length of boundary + name length.

    3. If entry‚Äôs value is a File:

      1. Let file be entry‚Äôs value.

      2. Let filename length be the length of the result of escaping a multipart/form-data name given file‚Äôs name with encoding and isFilename set to true.

      3. Increase entry length by 29 + filename length + the length of file‚Äôs type + file‚Äôs size.

      Otherwise:

      1. Let value be entry‚Äôs value, with every occurrence of U+000D (CR) not followed by U+000A (LF), and every occurrence of U+000A (LF) not preceded by U+000D (CR), replaced by a string consisting of U+000D (CR) and U+000A (LF).

      2. Increase entry length by the length of the result of encoding value with encoding.

  4. Return length.

The multipart/form-data length is the length of the byte sequence that results from running the multipart/form-data serializer. This is provided as a separate algorithm to make it possible to use ahead of time, without the need to read any files from the filesystem before their contents are needed.

Do we need to define this in the spec? That seems way too error-prone. Can‚Äôt we simply take the length of the serialization in spec-land, with a note saying that UAs can optimize with file.size?

The multipart/form-data serializer takes an entry list entries and a boundary string boundary, with an optional encoding encoding (default UTF-8), and then runs these steps:

  1. Set encoding to the result of getting an output encoding from encoding.

  2. Let output be the empty byte sequence.

  3. For each entry in entries:

    1. Let name be entry‚Äôs name.

    2. Append `--`, followed by boundary, followed by 0x0D 0x0A (CR LF), to output.

    3. Append `Content-Disposition: form-data; name="`, followed by the result of escaping a multipart/form-data name given entry‚Äôs name and encoding, followed by 0x22 ("), to output.

    4. If entry‚Äôs value is a string:

      1. Append 0x0D 0x0A 0x0D 0x0A (CR LF CR LF) to output.

      2. Let value be entry‚Äôs value.

      3. Replace every occurrence of U+000D (CR) not followed by U+000A (LF), and every occurrence of U+000A (LF) not preceded by U+000D (CR), in value, by a string consisting of U+000D (CR) and U+000A (LF).

      4. Append the result of encoding value with encoding, converted to a byte sequence, to output.

      Otherwise:

      1. Let file be entry‚Äôs value.

      2. Assert: file is a File.

      3. Append `; filename="`, followed by the result of escaping a multipart/form-data name given file‚Äôs name with encoding and isFilename set to true, followed by 0x22 0x0D 0x0A (" CR LF), to output.

      4. Append `Content-Type: `, followed by the result of isomorphic encoding file‚Äôs type, to output.

      5. Append 0x0D 0x0A 0x0D 0x0A (CR LF CR LF) to output.

      6. Append file‚Äôs underlying byte sequence to output.

    5. Append 0x0D 0x0A (CR LF) to output.

  4. Append `--`, followed by boundary, followed by `--`, followed by 0x0D 0x0A (CR LF), to output.

  5. Return output.

Or should this enqueue chunks into a ReadableStream instead?

This matches the behavior Gecko will have once bug 1686765 is closed, modulo the non-standard handling of directories. I haven‚Äôt checked Blink/WebKit.

To escape a multipart/form-data name with a string name, an optional encoding encoding (default UTF-8) and an optional boolean isFilename (default false):

  1. If isFilename is true:

    1. Set name to the result of converting name into a scalar value string.

    Otherwise:

    1. Replace every occurrence of U+000D (CR) not followed by U+000A (LF), and every occurrence of U+000A (LF) not preceded by U+000D (CR), in name, by a string consisting of U+000D (CR) and U+000A (LF).

  2. Let encoded be the result of encoding name with encoding, converted to a byte sequence.

  3. Replace every 0x0A (LF) bytes in encoded with the byte sequence `%0A`, 0x0D (CR) with `%0D` and 0x22 (") with `%22`.

  4. Return encoded.
